# v0.3 Spec — Rendering Foundation

**Date:** 2026-03-20
**Goal:** Complete the rendering pipeline to world-class quality. Every capability built to completion — general-purpose, extractable, second to none. Then iterate on visual design collaboratively.

**Guiding principle:** Build complete foundations before moving up the stack. Each capability should be a standalone, best-in-class library component, not minimum scaffolding wired to one use case.

**Design reference:** `design/philosophy.md` governs all decisions. Uncertainty isolated behind interfaces. Simple connective tissue, leaf-node complexity.

---

## Scope

### In scope

1. **Animation system** — timeline, easing, spring physics, property interpolation
2. **Smooth scroll** — sub-pixel offsets, momentum, velocity decay, inertia
3. **Proportional text layout** — line breaking, word wrap, mixed font metrics, paragraph layout
4. **Arbitrary path clipping** — clip to any bezier contour, not just rounded rects
5. **Backdrop blur / frosted glass** — blur content behind a translucent surface
6. **Pointer cursor** — visual cursor on screen tracking mouse/trackpad position
7. **Document switching** — fix Ctrl+Tab image viewer (migration gap)
8. **Text editor completeness** — navigation, selection, key combos
9. **Visual polish** — collaborative session: fonts, spacing, colors, effects

### Out of scope (later milestones)

- Multi-display (keep interfaces clean for future addition)
- BiDi / complex scripts
- Video / animated media (v0.6)
- Rich inline text / multi-style runs (v0.5)
- Undo/redo (needs COW filesystem, v0.4)
- System clipboard
- Full color design system

## Architecture

All new capabilities are **libraries** behind clean interfaces. The render services (`cpu-render`, `virgil-render`) consume them. Core produces scene graphs that use them. No capability is wired to only one consumer.

```text
Core (animation, layout, scene building)
  → Scene Graph (shared memory, triple-buffered)
    → Render Service (tree walk, clipping, blur, compositing, present)
      → Display
```

New libraries / library extensions:

- `libraries/animation/` — standalone animation engine (no dependencies on scene, render, or core)
- `libraries/layout/` — proportional text layout engine (depends on fonts for metrics)
- `libraries/render/` — extended with path clipping and backdrop blur
- `libraries/scene/` — extended with clip path and backdrop blur fields on Node
- `libraries/drawing/` — extended with stencil/mask operations if needed

### Node struct growth (cross-cutting)

Adding `clip_path: DataRef` (8 bytes) and `backdrop_blur_radius: u8` (1 byte) to Node. Current Node is exactly 120 bytes with 3 bytes of padding (`_pad: u8` + `_shadow_pad: [u8; 2]`).

**Plan:** Repurpose `_pad` (1 byte) for `backdrop_blur_radius: u8`. Add `clip_path: DataRef` (8 bytes) as a new field. Add `_reserved: [u8; 7]` for future use. Node grows from 120 to **136 bytes** (17 × 8, aligned).

**Growth policy:** Node grows in 8-byte increments to maintain aarch64 cache-line friendliness. The `_reserved` field provides 7 bytes of headroom for future fields (e.g., `z_order: i16`, `accessibility_id: u16`, layout hints) without another size bump. When `_reserved` is exhausted, grow by another 8 bytes.

**Impact:** This changes `NODES_OFFSET`, `DATA_OFFSET`, `SCENE_SIZE`, and `TRIPLE_SCENE_SIZE` (all derived from `size_of::<Node>()`). Total scene memory increase: 512 nodes × 16 extra bytes × 3 buffers = 24 KB. Negligible. Requires a coordinated single-commit update to: scene library (size assertion, Node struct), core (no code changes needed — uses Node by field name), cpu-render (no code changes — uses Node by field name), virgil-render (same), init (TRIPLE_SCENE_SIZE used for shared memory allocation), and test expectations that reference scene layout constants.

This is a mechanical change — the fields are accessed by name, not offset. The compile-time size assertion catches any mismatch.

### Vectors everywhere

All visual content uses resolution-independent vector representations. No bitmaps for shapes, cursors, or icons. Specifically:

- Pointer cursor: `Content::Path` commands (arrow, I-beam), not bitmap sprites
- Icons: glyph cache (already vector), not rasterized bitmaps
- Clip shapes: path commands, rasterized to mask only at render time by the backend
- The only bitmap content is user-provided images (PNG decode → BGRA pixels)

## Phase 1 — Motion

### 1.1 Animation Library (`libraries/animation/`)

A standalone, general-purpose animation engine. No dependencies on scene graph, rendering, or core. Pure `no_std`, no `alloc` required for fixed-capacity usage.

**Core types:**

- `Animation` — a single property animation: start value, end value, duration, easing function, start time, state (running/completed/cancelled). Generic over property type.
- `Spring` — a spring physics simulation: target value, current value, velocity, stiffness, damping, mass. Tick-based update. Settles when velocity and displacement are below threshold.
- `Timeline` — manages a set of active animations. Fixed-capacity array of 32 slots (no heap allocation). `tick(now)` advances all animations, returns which ones changed. `start()`, `cancel()`, `is_active()`. 32 slots supports: cursor blink (1) + scroll animation (1) + pointer fade (1) + transition (1) + per-node property animations (up to 28 concurrent). Exceeding capacity returns an error; caller can cancel oldest non-essential animation.
- `EasingFn` — function pointer or enum for easing curves.

**Easing functions (complete set):**

Standard CSS easing curves (cubic bezier evaluation):

- `linear`
- `ease` (default: 0.25, 0.1, 0.25, 1.0)
- `ease_in` (0.42, 0, 1, 1)
- `ease_out` (0, 0, 0.58, 1)
- `ease_in_out` (0.42, 0, 0.58, 1)
- `cubic_bezier(x1, y1, x2, y2)` — arbitrary cubic bezier easing

Additional curves:

- `ease_in_quad`, `ease_out_quad`, `ease_in_out_quad`
- `ease_in_cubic`, `ease_out_cubic`, `ease_in_out_cubic`
- `ease_in_expo`, `ease_out_expo`, `ease_in_out_expo`
- `ease_in_back`, `ease_out_back`, `ease_in_out_back` (overshoot)
- `ease_in_elastic`, `ease_out_elastic` (oscillating overshoot)
- `ease_in_bounce`, `ease_out_bounce` (bouncing)
- `step_start`, `step_end` (discrete steps)

**Spring physics:**

Critically damped spring (like iOS/macOS):

- `Spring::new(stiffness, damping, mass)` with presets: `Spring::DEFAULT`, `Spring::SNAPPY`, `Spring::GENTLE`, `Spring::BOUNCY`
- `Spring::tick(dt) → (value, settled)` — advances simulation by `dt` seconds
- Configurable settle threshold
- Works for any `f32` property

**Interpolation:**

- `Lerp` trait: `fn lerp(a: Self, b: Self, t: f32) -> Self`
- Implementations for: `f32`, `i32`, `u8`, `Color` (interpolate in linearized space via `SRGB_TO_LINEAR` LUT, convert back to sRGB), `AffineTransform` (component-wise)

**Testing:** Comprehensive host-side tests. Easing curve accuracy (compare against reference implementations). Spring convergence (must settle within expected time). Timeline lifecycle (start, tick, complete, cancel). Edge cases (zero duration, negative time, NaN). Capacity overflow behavior.

### 1.2 Smooth Scroll

Integrate animation library into core's scroll handling.

**Scroll model migration (line-count → pixel-space):**

Currently `scroll_offset` is `u32` in CoreState representing a line count. It is converted to pixel space as `scroll_offset * line_h` before being passed as `scroll_y: i32` to layout functions.

Migration:

- `scroll_offset` becomes `scroll_offset: f32` in pixel space (not line count)
- All layout functions that accept `scroll_y: i32` migrate to `scroll_y: f32`
- `scroll_for_cursor()` returns pixel-space values instead of line counts
- `update_scroll_offset()` comparisons use epsilon-based float comparison (threshold: 0.5 physical pixels)
- Cursor-keeping-visible logic targets sub-pixel positions but snaps the final content_transform to avoid jitter when scroll settles

**Keyboard scroll:**

- Arrow keys (when at document top/bottom causing scroll): spring animation to target line position
- PgUp/PgDn: spring animation scrolling by viewport height

**Momentum scroll (keyboard only in v0.3):**

- Mouse wheel / trackpad scroll events are not yet wired (virtio-tablet provides absolute position, not scroll wheel). When mouse wheel support is added later, velocity feeds into the same momentum system.
- For now, momentum is triggered by repeated rapid key-scroll events

**Overscroll:**

- Maximum overscroll: 50 points past document bounds
- Spring-back animation when released (Spring::SNAPPY preset)

**Integration with content_transform:**

- Core sets `content_transform = AffineTransform::translate(0.0, -scroll_offset)` on the document container node
- The render service already handles fractional transforms
- Incremental update: scroll changes only update `content_transform` and mark the container dirty

### 1.3 Animated Cursor Blink

- Cursor opacity animates using the animation timeline
- Fade in/out (not abrupt toggle): 0.0 → 1.0 over ~150ms, hold ~500ms, 1.0 → 0.0 over ~150ms, hold ~300ms
- Any keystroke or click resets to fully visible and restarts the cycle
- Uses `ease_in_out` for opacity transitions

### 1.4 Animated Transitions (foundation)

With the animation library in place, add transition support to core:

- Document switching: fade transition (old scene fades out, new fades in) via root node opacity
- Scroll position changes: spring animation (not instant jump)
- Selection changes: smooth highlight appearance (opacity fade-in over ~100ms)

These are lightweight applications of the animation library — they prove the system works for real use cases beyond cursor blink.

## Phase 2 — Composition

### 2.1 Arbitrary Path Clipping

**Scene graph extension:**

- `Node` gains `clip_path: DataRef` — reference to serialized path commands in the data buffer (same format as `Content::Path`). `DataRef::EMPTY` (offset=0, length=0) means no clip.
- `clip_path` clips all children of the node, not just the node itself
- `corner_radius` remains as-is — the existing SDF-based fast path is preserved. `clip_path` is a separate, additional capability for arbitrary shapes. They do not overlap: `corner_radius` handles the common rounded-rect case efficiently, `clip_path` handles everything else.

**CpuBackend implementation:**

- Rasterize clip path to an 8bpp alpha mask (u8 per pixel) at the node's bounds. 8bpp rather than 1bpp for anti-aliased clip edges — same scanline rasterizer used for `Content::Path`.
- During child rendering, multiply destination alpha by mask value
- Mask stored in a separate clip mask cache (not the BGRA surface pool — different allocation size and lifetime). Fixed capacity: 16 cached masks, LRU eviction. Invalidated when clip path DataRef, node bounds, or transform changes.
- Memory budget: 16 masks × average 200×200 pixels × 1 byte = ~640 KB worst case. Acceptable.
- Supports any path expressible as MoveTo/LineTo/CubicTo/Close

**virgil-render implementation:**

- Use GPU stencil buffer: render clip path to stencil, enable stencil test for children
- Stencil push/pop for nested clips (increment/decrement stencil value)
- Same path format, different rasterization backend

**Testing:** Clip to circle, triangle, star, complex bezier outline. Nested clips. Clip + rounded corners (both active simultaneously). Clip + transforms. Anti-aliased clip edges. Verify both cpu-render and virgil-render produce matching results (screenshot comparison).

### 2.2 Backdrop Blur / Frosted Glass

**Scene graph extension:**

- `Node.backdrop_blur_radius: u8` replaces `Node._pad: u8` (0 = disabled, non-zero = blur radius in points)
- Effective pixel radius = `backdrop_blur_radius * scale_factor`, clamped to `MAX_BACKDROP_BLUR_PX` (32 physical pixels for CpuBackend, uncapped for GPU)

**CpuBackend implementation:**

- The existing `MAX_CPU_BLUR_RADIUS` is 16px. For backdrop blur, raise to 32px (still single-digit milliseconds with NEON SIMD on the typical backdrop region). Radii > 32px are clamped with no visual discontinuity (large blurs are perceptually similar past ~20px).
- Render the scene tree up to (but not including) the backdrop-blur node
- Extract the rectangular region behind the node from the framebuffer
- Apply existing Gaussian blur (already NEON-accelerated)
- Composite the blurred region back
- Then render the node (typically translucent) on top
- Cache the blurred backdrop keyed on: node bounds + backdrop*blur_radius + underlying content hash. Skip re-blur when only the overlay content changed (check dirty bitmap — if no dirty nodes are \_behind* the blur surface, cache hit).

**virgil-render implementation:**

- Requires new shader programs for the blur pass: horizontal and vertical Gaussian fragment shaders (two passes). These are new additions to `shaders.rs`.
- Render scene to intermediate texture (render-to-texture) up to the blur node
- Blur pass: bind intermediate texture, draw fullscreen quad through horizontal blur shader, then vertical blur shader
- Composite blur result + node content onto final framebuffer
- This is a significant addition to the GPU pipeline (~200-300 lines of new shader/pipeline code) but follows established patterns from the existing textured-quad rendering.

**Testing:** Blur over text content. Blur over images. Blur radius variations (small, medium, at-cap). Cache effectiveness: verify only first frame is slow, subsequent frames use cache. Visual verification via screenshots.

### 2.3 Pointer Cursor

**Depends on:** Phase 1 animation library (for fade in/out).

**Scene graph:**

- Top-level pointer node added to scene tree at highest z-order
- `Content::Path` with arrow cursor shape (same procedural arrow as before, but now in the scene graph)
- Position: `(mouse_x, mouse_y)` in points, updated via incremental path (like cursor blink — position-only change, zero allocation)

**Core integration:**

- `MSG_POINTER_ABS` updates position → marks pointer node dirty → incremental publish
- New incremental update path: `update_pointer()` — moves pointer node, no layout recalculation
- Pointer hidden when no mouse events received for 3 seconds (fade out via animation timeline, `ease_out` over 300ms)
- Pointer shown immediately on any mouse event (instant opacity=255, no fade-in)

**Cursor shapes:**

- Arrow (default) — standard pointer
- Text (I-beam) — over text content area (hit test against document bounds)
- Shape switching based on `mouse_y` position: above title bar height → arrow, within document area → I-beam
- Each shape is a pre-defined set of path commands stored as constants

**Testing:** Pointer appears on mouse event. Position tracks correctly. I-beam in text area, arrow elsewhere. Hide after timeout. Show on movement.

### 2.4 Document Switching Fix

**Root cause:** `build_editor_scene()` ignores `image_mode`. Image viewer scene was never ported from old compositor to core.

**Fix:**

- Add `image_mode: bool` parameter to `build_editor_scene()` and `build_full_scene()`
- When `image_mode == true`:
  - Content area shows centered PNG image
  - Title bar shows "Image viewer | WxH px" instead of editor info
  - No cursor, no selection nodes
  - Chrome frame (shadows, rounded corners, title bar) stays the same
- Ctrl+Tab transition uses fade animation from Phase 1
- Pass `state.image_mode` from main.rs event loop

**Image data handling:**

- The decoded BGRA image data lives in shared memory mapped by init (VA provided via `MSG_IMAGE_CONFIG`). This data is too large for the 64 KB scene data buffer (a 1024x768 BGRA image = 3 MB).
- Solution: `Content::Image` gets a new variant or flag indicating "external VA reference." When `data.offset == u32::MAX`, the render service uses its pre-mapped image VA (received via compositor config) instead of the scene data buffer. The `src_width` and `src_height` fields tell the renderer the image dimensions.
- This pattern (scene graph carries metadata, render service holds the actual data) is consistent with how glyph rendering works: the scene graph has glyph IDs, the render service has the glyph cache.

**Testing:** Ctrl+Tab switches to image view. Image centered and correctly sized. Title bar updates. Ctrl+Tab back returns to editor. Screenshot verification for both modes.

## Phase 3 — Text & Interaction

### 3.1 Proportional Text Layout

**New library: `libraries/layout/`**

A general-purpose text layout engine. Depends only on font metrics (not on specific font library internals). Pure `no_std` + `alloc`.

**Core capabilities:**

- `layout_line(glyphs: &[ShapedGlyph], max_width: f32) -> LineLayout` — position shaped glyphs within a line, handle alignment (left, center, right)
- `layout_paragraph(text: &str, font_metrics: &dyn FontMetrics, max_width: f32, alignment: Alignment) -> ParagraphLayout` — full paragraph layout with word wrap
- Word-wrap line breaking: break at spaces, hyphens, and CJK boundaries. The interface accepts a `LineBreaker` trait so a full UAX #14 implementation can be swapped in later without changing the layout engine. Default implementation: break at ASCII spaces and hyphens.
- `ParagraphLayout` contains: line count, per-line glyph positions, per-line bounding boxes, total height
- `FontMetrics` trait: `fn advance_width(&self, glyph_id: u16) -> f32`, `fn ascender(&self) -> f32`, `fn descender(&self) -> f32`, `fn line_gap(&self) -> f32`, `fn space_width(&self) -> f32`

**Integration with core:**

- Chrome text (title bar, status bar, labels) uses `layout_line()` with proportional font metrics
- Document text continues to use `layout_mono_lines()` (monospace, existing code)
- The interface is the same — both produce positioned glyph runs that become `Content::Glyphs` nodes

**Testing:** Single-line layout with varying glyph widths. Word wrap at width boundary. Alignment (left, center, right). Empty text. Very long words (no break opportunity — overflow). Mixed-width characters. Paragraph with multiple lines. LineBreaker trait with custom implementation.

### 3.2 Text Editor Key Combinations

**Responsibility split:** The existing architecture has core handling system-level keys (Ctrl+Tab) and forwarding raw key events to the text editor process. The editor process handles document-level editing (character insertion, deletion, cursor movement). This split is preserved.

**Editor-side (text-editor process):** All navigation and editing that operates on document content. The editor has read-only access to the document buffer and knows the viewport dimensions from `EditorConfig`.

**Key binding model:** macOS-style. QEMU on macOS maps Cmd to Super (evdev `KEY_LEFTMETA`/`KEY_RIGHTMETA`). The virtio-input driver must track modifier state (Shift, Alt/Option, Super/Cmd, Caps Lock) and forward it with key events.

**Navigation:**

- Up/Down arrow: move cursor to same visual column on adjacent line. Uses `byte_to_line_col()` for current position, `line_col_to_byte()` for target. Remembers "goal column" across vertical moves (sticky column).
- Super+Left (Cmd+Left): move to start of current line
- Super+Right (Cmd+Right): move to end of current line
- Alt+Left (Opt+Left): move to previous word boundary (scan backward for whitespace→non-whitespace transition)
- Alt+Right (Opt+Right): move to next word boundary (scan forward)
- Super+Up (Cmd+Up): move to start of document
- Super+Down (Cmd+Down): move to end of document
- Home: move to start of current line (alternative to Super+Left)
- End: move to end of current line (alternative to Super+Right)
- PgUp: move cursor up by viewport line count (editor knows viewport height from config)
- PgDn: move cursor down by viewport line count

**Selection (Shift + any navigation key extends selection):**

- Shift+Up/Down: extend selection vertically
- Shift+Super+Left/Right: select to line start/end
- Shift+Alt+Left/Right: select word
- Super+A (Cmd+A): select all (anchor=0, cursor=end)

**Editing:**

- Tab: insert 4 spaces at cursor position
- Shift+Tab: remove up to 4 leading spaces from current line (dedent)
- Delete key (Fn+Backspace on Mac): delete character after cursor (forward delete)
- Alt+Backspace (Opt+Backspace): delete word backward
- Alt+Delete (Opt+Fn+Backspace): delete word forward

**Character input:**

- Shift+key: uppercase character (handled by evdev shift state in virtio-input driver)
- Caps Lock: toggle uppercase mode. Requires tracking LED state from evdev `EV_LED` events. The virtio-input driver tracks caps lock state and applies it to key-to-character translation.

**Core-side:** Ctrl+Tab (document switching), pointer event processing, click hit-testing. Core forwards all other key events to the editor unchanged.

**Core-side (mouse click processing):**

- Single click: position cursor (existing behavior)
- Double-click: select word at click position. Word boundaries: scan to nearest whitespace in both directions.
- Triple-click: select entire line at click position.
- Click counting: track last click time and position. Double-click window: 400ms and within 4px of previous click. Triple-click: 400ms after double-click, same position threshold. Beyond triple, cycle back to single-click behavior.

**Unicode input:**

- Non-ASCII characters from keyboard input forwarded as UTF-8 bytes
- The text buffer is already UTF-8 — insertion is byte-level, cursor movement respects codepoint boundaries (skip continuation bytes 0x80-0xBF)

## Build Order

Foundation-first. Each phase completed fully before the next begins.

1. **Phase 1 — Motion:** Animation library (complete, general-purpose, fully tested) → smooth scroll (with scroll model migration) → cursor blink → transitions
2. **Phase 2 — Composition:** Node struct growth (single coordinated commit) → path clipping (complete, both backends) → backdrop blur (complete, both backends) → pointer cursor → document switching fix
3. **Phase 3 — Text & Interaction:** Proportional layout library (complete, fully tested) → editor key combos (all listed) → mouse click enhancements (double/triple-click)
4. **Phase 4 — Visual Polish:** Collaborative session

Within each phase: library first (tested independently), then integration into core/render services, then visual verification.

- All existing tests pass (~1,943 system + 21 prototype)
- Each new capability has comprehensive host-side tests
- Visual verification via QEMU screenshots at each phase boundary
- Animation: frame timing instrumented via `sys::counter()` — verify frame intervals are consistent (no >2x variance). Take timed screenshot sequences showing smooth animation progress.
- Composition: backdrop blur visible behind translucent chrome panels
